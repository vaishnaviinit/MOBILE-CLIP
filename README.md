# MobileCLIP Phishing Screenshot Classifier

A production-grade binary image classifier that detects phishing websites from screenshots using MobileCLIP2-S2. This is one component of a larger phishing detection platform — it handles visual classification only. A separate repository will train a URL-feature model; both will be fused in an ensemble.

## Overview

The model learns to distinguish phishing pages from legitimate ones by visual appearance alone — page layout, typography, button placement, login forms, logo usage, and UI consistency. It deliberately does **not** see the URL, page source, or any text content. This forces it to generalise on visual patterns rather than memorise domains.

**Model:** MobileCLIP2-S2 (36M parameters, 256×256 input, 512-dim embeddings, pretrained on DFN-2B)  
**Task:** Binary classification — `legitimate` (0) vs `phishing` (1)  
**Security priority:** Minimise False Negatives (phishing missed) over False Positives (legit flagged)

## Where This Fits

```
User submits URL
       |
       v
Screenshot Generation Pipeline   <-- separate repo
       |
       v
 Website Screenshot               dataset/legitimate/  dataset/phishing/
       |
       v
  THIS REPOSITORY
  MobileCLIP2-S2
  Binary Classifier + EMA
       |
       +---> P(phishing)   }
       |                   }  --> Future Ensemble Model --> Final Verdict
       +---> 512-dim embed }         (+ URL features)
```

The `extract_features()` method on the backbone is the contract with the future ensemble. It returns L2-normalised 512-dim embeddings that can be concatenated with URL features and passed to a meta-learner.

## Dataset Setup

The dataset is generated by a separate screenshot pipeline. Place it in the `dataset/` folder with this exact structure:

```
dataset/
    legitimate/
        amazon/
            homepage.png
            signup.png
        microsoft/
            homepage.png
        google/
            homepage.png
            login.png
        ...      <- any number of brands, any number of screenshots per brand
    phishing/
        generic/
            paypal_fake_001.png
            microsoft_phish_042.png
            ...
```

**Rules:**
- Legitimate images go under `dataset/legitimate/<brand_name>/`
- Phishing images go under `dataset/phishing/generic/` (or any subfolder)
- Supported formats: `.png`, `.jpg`, `.jpeg`, `.webp`, `.bmp`
- The code discovers all images recursively — no config changes needed when adding more data

**Current dataset stats:**

```
Total:      3,485 images
Legitimate: 2,935 (1,332 brands)
Phishing:     550 (1 folder)
Imbalance:  5.34:1 (legitimate : phishing)
```

The imbalance is handled automatically via `WeightedRandomSampler` and `FocalLoss`.

## Training

### Basic training

```bash
python scripts/train.py
```

All hyperparameters are read from `configs/config.yaml`. On first run, MobileCLIP2-S2 weights are downloaded from HuggingFace (~140 MB) and cached at `~/.cache/huggingface/hub/`.

### Resume interrupted training

```bash
python scripts/train.py --resume outputs/checkpoints/last_checkpoint.pt
```

This restores epoch number, optimizer state, scheduler state, early stopping counter, and the best metric seen so far — training continues exactly where it left off.

### Override config values on the command line

```bash
# Use a bigger batch size (if you have more GPU VRAM)
python scripts/train.py --override training.batch_size=64

# Train for more epochs
python scripts/train.py --override training.epochs=100

# Change the learning rate
python scripts/train.py --override training.optimizer.lr_head=5e-4

# Multiple overrides at once
python scripts/train.py --override training.batch_size=64 training.epochs=100 model.dropout=0.4
```

### Monitor training in real time

```bash
# Follow the log file
tail -f outputs/logs/training.log

# Watch metrics per epoch (after at least 1 epoch completes)
python -c "import pandas as pd; print(pd.read_csv('outputs/logs/training_logs.csv').to_string())"
```

### What happens during training

**Phase 1 (epochs 1–5):** The MobileCLIP2 backbone is frozen. Only the small classification head (512→256→2) is trained at `lr_head=1e-3`. This warms up the head before touching the pretrained features.

**Phase 2 (epochs 6–50):** The backbone unfreezes. Two learning rates are used simultaneously:
- Backbone: `lr_backbone=1e-5` (100× smaller — gentle nudging of pretrained features)
- Head: `lr_head_phase2=1e-4` (10× smaller than Phase 1 — already adapted)

The scheduler uses 2 warmup epochs then cosine decay to `min_lr=1e-7`.

**EMA (Exponential Moving Average):** After every optimizer step, a shadow copy of all weights is updated:
```
shadow = 0.999 × shadow + 0.001 × current_weights
```
Validation always runs with EMA weights (swapped in temporarily). `best_model.pt` saves the EMA weights. `last_checkpoint.pt` saves the live training weights (needed for correct resume). EMA consistently gives 1–3% better generalisation at zero extra training cost.

**Early stopping** watches `val_f2` (F2 score computed with EMA weights). Training stops if `val_f2` does not improve for 10 consecutive epochs.

### Training outputs

```
outputs/
    checkpoints/
        best_model.pt        <- EMA weights, saved when val_f2 improves (use for inference)
        last_checkpoint.pt   <- training weights, saved every epoch (use for resume)
    logs/
        training.log         <- full structured log
        training_logs.csv    <- one row per epoch (for plotting)
```

> **Why two checkpoint files?** `best_model.pt` contains the smoothed EMA weights — these give the best inference quality. `last_checkpoint.pt` contains the raw training weights with optimizer state — these are what `--resume` needs to continue training correctly. Never use `last_checkpoint.pt` for inference.

## Evaluation

Run the full evaluation suite on the held-out test set:

```bash
python scripts/evaluate.py
```

### Options

```bash
# Use a specific checkpoint
python scripts/evaluate.py --checkpoint outputs/checkpoints/best_model.pt

# Override the decision threshold
python scripts/evaluate.py --threshold 0.35

# Change threshold selection strategy (default: f2)
python scripts/evaluate.py --strategy f1        # balanced precision/recall
python scripts/evaluate.py --strategy f2        # recall-heavy (recommended)
python scripts/evaluate.py --strategy min_fnr   # guarantee >=95% phishing caught

# Also generate GradCAM heatmaps for misclassified samples
python scripts/evaluate.py --gradcam
```

### What evaluation produces

```
outputs/predictions/
    evaluation_report.json      <- recommended threshold + all metrics
    predictions.json            <- per-sample predictions with probabilities
    false_negatives.json        <- phishing pages that were missed (sorted by danger)
    false_positives.json        <- legitimate pages incorrectly flagged
    classification_report.txt   <- precision/recall/F1 per class (sklearn format)
outputs/visualizations/
    confusion_matrix.png
    roc_curve.png
    pr_curve.png
    threshold_sweep.png
```

### Understanding the threshold output

The evaluator sweeps 181 thresholds and reports three operating points:

| Operating Point | Description | When to use |
|---|---|---|
| **Max F1** | Best balance of precision and recall | General purpose |
| **Max F2** | Maximises recall (catches more phishing) | **Default deployment** |
| **Min FNR** | Highest threshold where FNR ≤ 5% | Maximum security mode |

After running evaluate.py, `infer.py` automatically uses the F2-optimised threshold from `evaluation_report.json`.

## Inference

### Classify a single screenshot

```bash
python scripts/infer.py --image path/to/screenshot.png
```

**Output:**
```json
{
  "image": "path/to/screenshot.png",
  "predicted_class": "phishing",
  "confidence": 0.9412,
  "phishing_probability": 0.9412,
  "legitimate_probability": 0.0588,
  "threshold_used": 0.350,
  "threshold_source": "evaluation_report.json (strategy: f2)",
  "inference_time_ms": 18.4,
  "tta_n": 1,
  "model": "MobileCLIP2-S2",
  "embedding_dim": 512
}
```

### Options

```bash
# Override the threshold
python scripts/infer.py --image screenshot.png --threshold 0.25

# Test-Time Augmentation: average over 6 views (1 center + 5 random crops)
# Improves recall by 2-5% at the cost of 6x inference time — recommended for production
python scripts/infer.py --image screenshot.png --tta

# More views for borderline/uncertain cases
python scripts/infer.py --image screenshot.png --tta --tta-n 10

# Human-readable output instead of JSON
python scripts/infer.py --image screenshot.png --no-json

# Extract only the 512-dim embedding (for ensemble downstream use)
python scripts/infer.py --image screenshot.png --embed-only

# Generate a GradCAM heatmap showing what the model looked at
python scripts/infer.py --image screenshot.png --gradcam
```

### TTA output (with --tta)

```json
{
  "predicted_class": "phishing",
  "phishing_probability": 0.8931,
  "tta_n": 6,
  "tta_individual_probs": [0.8812, 0.8945, 0.9103, 0.8654, 0.9021, 0.9072],
  "inference_time_ms": 112.4
}
```

`tta_individual_probs` shows each view's raw probability — useful for debugging borderline predictions. If all views agree, the model is confident. If they spread wide, the image is genuinely ambiguous.

### Threshold priority

The threshold is resolved in this order:
1. `--threshold` CLI argument (highest priority)
2. `outputs/predictions/evaluation_report.json` → `recommended_threshold` (set by evaluate.py)
3. `configs/config.yaml` → `evaluation.threshold` (fallback, default 0.5)

**Always run `evaluate.py` after training** to compute the optimal threshold before using `infer.py` in production.

## Configuration Reference

All settings live in `configs/config.yaml`. Key sections:

### `model`

| Key | Default | Description |
|---|---|---|
| `backbone` | `MobileCLIP2-S2` | OpenCLIP model name |
| `pretrained` | `dfndr2b` | Pretrained checkpoint tag (DFN-2B dataset) |
| `embedding_dim` | `512` | Feature vector size |
| `freeze_backbone_epochs` | `5` | How many epochs to keep backbone frozen |
| `dropout` | `0.3` | Dropout in classifier head |
| `hidden_dim` | `256` | Hidden units in classifier head |

### `training`

| Key | Default | Description |
|---|---|---|
| `batch_size` | `32` | Samples per batch (increase for GPU with more VRAM) |
| `epochs` | `50` | Max training epochs |
| `device` | `auto` | `auto` / `cuda` / `cpu` / `mps` |
| `mixed_precision` | `true` | FP16 autocast (only on CUDA) |
| `gradient_clip` | `1.0` | Max gradient norm |
| `ema_decay` | `0.999` | EMA smoothing factor. Set to `0` to disable EMA. Higher = slower EMA adaptation |

### `training.optimizer`

| Key | Default | Description |
|---|---|---|
| `lr_head` | `1e-3` | Head LR in Phase 1 (linear probe) |
| `lr_head_phase2` | `1e-4` | Head LR in Phase 2 (fine-tune) |
| `lr_backbone` | `1e-5` | Backbone LR in Phase 2 |
| `weight_decay` | `0.01` | AdamW weight decay |

### `training.early_stopping`

| Key | Default | Description |
|---|---|---|
| `monitor` | `val_f2` | Metric to track for improvement |
| `patience` | `10` | Epochs without improvement before stopping |

### `evaluation`

| Key | Default | Description |
|---|---|---|
| `threshold_strategy` | `f2` | `f1`, `f2`, or `min_fnr` |
| `min_fnr_target` | `0.05` | Max allowed FNR for `min_fnr` strategy |

### `data`

| Key | Default | Description |
|---|---|---|
| `num_workers` | `0` | DataLoader workers. Set to 4 on Linux/Mac with GPU |
| `pin_memory` | `false` | Set to `true` when using CUDA with `num_workers > 0` |
| `image_size` | `256` | Input resolution (must match backbone) |

## Outputs

### Checkpoints

Two checkpoint files are saved with different purposes:

**`best_model.pt`** — for inference and evaluation:
```
{
  "epoch":            17,
  "model_state":      {...},   <- EMA weights (smoother, better generalisation)
  "ema_weights_used": true,    <- flag confirming EMA weights are stored
  "metrics":          {"val_f2": 0.91, "val_recall": 0.94, ...},
  "ema":              {...},   <- EMA shadow weights (for reference)
  ...
}
```

**`last_checkpoint.pt`** — for resume only:
```
{
  "epoch":            17,
  "model_state":      {...},   <- live training weights
  "optimizer_state":  {...},   <- AdamW momentum buffers (needed for correct resume)
  "scheduler_state":  {...},   <- LR schedule position
  "scaler_state":     {...},   <- AMP GradScaler state
  "phase":            2,       <- which training phase we were in
  "ema":              {...},   <- EMA shadow weights (restored on resume)
  "early_stopping":   {"counter": 3, "best_value": 0.91},
  "model_checkpoint": {"best_value": 0.91}
}
```

### Metrics logged per epoch (training_logs.csv)

`epoch`, `train_loss`, `train_acc`, `val_loss`, `val_acc`, `val_precision`, `val_recall`, `val_f1`, `val_f2`, `val_roc_auc`, `lr_head_decay`, `lr_head_no_decay`

## Troubleshooting

### `Pretrained value 'dfndr2b' is not a known tag`

Your `open_clip_torch` version is too old. MobileCLIP2 requires open_clip_torch >= 2.24.
```bash
pip install --upgrade open_clip_torch
```

### `CUDA out of memory`

Reduce batch size:
```bash
python scripts/train.py --override training.batch_size=16
```

### Training is very slow on CPU

Expected. See [GPU Setup](#gpu-setup). You can also reduce the model for faster experimentation:
```bash
python scripts/train.py --override model.backbone=MobileCLIP-S1
```

### `FileNotFoundError: dataset/legitimate not found`

Check your dataset folder is named `dataset/` and sits inside the project root (`G:\mobile_clip\dataset\`). See [Dataset Setup](#dataset-setup).

### `No improvement in val_f2 for 10/10 epochs`

Early stopping triggered. Either:
- Training converged (check `val_f2` in training_logs.csv — if it's high, training is done)
- The model didn't learn (check `val_recall` — if it's near 0 or 1 constantly, there may be a data or config issue)

### Resume failed with `optimizer state mismatch`

This happens if you changed the model config between runs. Either:
- Start fresh: `python scripts/train.py` (no `--resume`)
- Or keep the same config as the original run

### MobileCLIP2-S2 checkpoint download is slow

The weights (~140 MB) cache at `~/.cache/huggingface/hub/`. On subsequent runs they load from cache instantly. If download repeatedly fails, manually download from HuggingFace and set `HF_HUB_OFFLINE=1`.

### `best_model.pt` gives different results than during training

Expected — `best_model.pt` stores EMA weights which are slightly different from the training weights at that exact epoch. EMA weights generalise better and this difference is intentional. Use `last_checkpoint.pt` only if you need to continue training.
